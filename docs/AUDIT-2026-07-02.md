## 3. Invariant verification

- **`#![forbid(unsafe_code)]`** present at the crate root; zero `unsafe`
  anywhere in the tree.
- **Never-mis-correct:** the decoder pipeline applies all five defenses
  required by the spec — Λ-degree bound, Chien root-count/degree consistency
  (which also absorbs out-of-range "phantom" roots of shortened codes via the
  `[0, n)` bound), zero Forney magnitude rejection, and mandatory
  post-correction syndrome verification as the primary defense. Every
  inconsistent branch returns `RsError::Uncorrectable`; no path can return
  plausible-but-wrong bytes.
- **Checked arithmetic:** `encoded_len` and the structural validation in
  `decode_blocks` use `checked_add`/`checked_mul`; `original_len =
  usize::MAX` maps to `InvalidInput` without overflow. The one unchecked
  product (`blocks * data_len` for the output reservation) is bounded by
  `encoded.len()` and guarded by the preceding `checked_mul`.
- **Bounded memory-exhaustion DoS:** all input-scaled allocations
  (`encode`, `decode`, framed paths) go through `try_reserve` and fail as
  `InvalidInput` instead of aborting; per-block scratch is constant-sized
  (≤ 255 bytes).
- **Pinned convention:** field polynomial `0x187`, conventional basis,
  `α = 0x02`, `FCR = 112` as named constants; the FCR-dependent Forney factor
  `X^(1−FCR)` is present and documented; KATs pin the convention against an
  independent `reedsolo` reference for both RS(255,223) and RS(15,11,4).
- **Zero runtime dependencies;** the self-contained CRC-32/IEEE is validated
  against the standard check value (`crc32(b"123456789") == 0xCBF43926`).
- **Documented footguns:** the silent subset-roots parameter-mismatch family
  is documented in `decode`'s rustdoc with the worked example required by the
  spec; the framed path closes it (`InvalidInput` on mismatch, covered by
  tests).
- **API contract:** the frozen v0.1.0 public surface is implemented exactly;
  all public items carry rustdoc; all source files carry the
  Author/Version/Date header.

## 4. Codec mathematics review

Manual line-by-line review of `gf256`, `poly`, `encode`, and `decode`:

- The compile-time `const` EXP/LOG tables are correct; the doubled EXP table
  (512 entries) removes the modulo from `mul` (maximum index 508 < 512).
- `inv`/`div`/`pow` handle zero operands fail-loud (`assert!` in all builds,
  documented as unreachable internal preconditions; 200k+ fuzz executions
  corroborate unreachability).
- The Berlekamp-Massey implementation is genuinely inversionless (no GF
  inverse in the iteration loop) and KAT-pinned.
- The potential slice underflow in `poly::remainder` is unreachable for
  every geometry `ReedSolomon::new` admits (`dividend.len() = data_len +
  parity_len ≥ divisor.len()` always holds).
- The three fuzz targets (`decode`, `encode`, `decode_framed`) match the
  spec's robustness bar; prior campaign: 200k+ executions, zero crashes.

## 5. Findings and remediation

All findings were metadata/style level — none touched logic, security,
structure, or performance. All three were remediated on 2026-07-02:

| # | Finding | Severity | Remediation |
|---|---------|----------|-------------|
| 1 | MSRV undeclared: `usize::is_multiple_of` (`src/decode.rs`) requires Rust ≥ 1.87, CI pins 1.91, but `Cargo.toml` had no `rust-version`, so older toolchains failed with a raw compile error instead of a clear cargo message | Metadata | `rust-version = "1.87"` added and **verified by compiling the full crate (all targets, dev-deps included) with a freshly installed 1.87 toolchain** — commit `47fff52` |
| 2 | `Cargo.lock` untracked: current Rust guidance recommends committing lockfiles (libraries included) so CI and `cargo audit` are reproducible | Optional | Root `Cargo.lock` and `fuzz/Cargo.lock` (standalone workspace) removed from `.gitignore` and committed — commit `73a57f5` |
| 3 | Style: `mod frame;` was the only module declaration in `src/lib.rs` without an explicit `pub(crate)`; effective visibility identical, but inconsistent on read | Style | Aligned to `pub(crate) mod frame;` — commit `ea9e048` |

The full gate suite (§2) was re-run and confirmed green after each
remediation commit.

## 6. Conclusion

The crate satisfies its specification and its critical invariant (fail-loud,
never mis-correct) with defense-in-depth, carries no third-party runtime
dependencies, and passes every quality gate. With the three metadata/style
findings remediated, no known issues remain. The `rust-version` declaration
is publishable metadata suitable for a future `0.1.1` release.
